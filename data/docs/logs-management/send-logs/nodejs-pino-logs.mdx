## Overview

This guide demonstrates how to configure Pino to send logs directly to SigNoz Cloud or Self-hosted SigNoz without using OpenTelemetry collectors. Following this documentation will enable you to stream your Node.js application logs directly to your SigNoz dashboard.

* Configure pino logger to send logs to SigNoz Cloud

## Prerequisites


    - Ingestion Key


## Required Dependencies

Install the necessary packages:

```bash
npm i pino-opentelemetry-transport
```

## Setup

### Step 1. Configure Pino Logger

```jsx
//tracing.js
const pino = require('pino')

const transport = pino.transport({
  target: 'pino-opentelemetry-transport'
})

const logger = pino(transport)

module.exports = logger
```

### Step 2. Configure Application Logging

```jsx
//app.js
const logger = require('./tracing');

// Basic logging
logger.info('Hello, World! from pino');

// Structured logging with metadata
logger.info('User authentication', {
    userId: 'user123',
    action: 'login',
    timestamp: new Date().toISOString()
});

// Error handling example
try {
    throw new Error('Something went wrong');
} catch (error) {
    logger.error({
        msg: 'Error occurred',
        error: error.message,
        stack: error.stack
    });
}
```

### Step 3. Run Your Application

Execute your application with OpenTelemetry Environment variables:

```jsx
OTEL_EXPORTER_OTLP_HEADERS="signoz-ingestion-key=<your-ingestion-key>" OTEL_EXPORTER_OTLP_LOGS_ENDPOINT=https://ingest.<region>.signoz.cloud:443/v1/logs OTEL_RESOURCE_ATTRIBUTES="service.name=<service_name>,service.version=1.2.3" node -r ./tracing.js app.js
```
- Set the <region> to match your SigNoz Cloud [region](https://signoz.io/docs/ingestion/signoz-cloud/overview/#endpoint)
- Replace <your-ingestion-key> with your SigNoz [ingestion key](https://signoz.io/docs/ingestion/signoz-cloud/keys/).
- <service_name> is name of your service
## Usage Examples

### Basic Logging

```jsx
logger.info('Server started on port 3000');

logger.error('Failed to process request');
```

- Monitor logs in [real-time](https://signoz.io/docs/product-features/logs-explorer/#live-view)
- Filter logs by level and custom attributes
- Search through log contents
- Create alerts based on log patterns
- Visualize log trends and patterns

If logs are not appearing in SigNoz:

1. Verify your Ingestion Keys and endpoint URL
2. Ensure the transport is properly configured
3. Review SigNoz dashboard filters
4. Check Pino log level settings

## Setup

### Step 1. Configure Pino Logger

```jsx
//tracing.js
const pino = require('pino')

const transport = pino.transport({
  target: 'pino-opentelemetry-transport'
})

const logger = pino(transport)

module.exports = logger
```

### Step 2. Configure Application Logging

```jsx
//app.js
const logger = require('./tracing');

// Basic logging
logger.info('Logs from pino');

// Structured logging with metadata
logger.info('User authentication', {
    userId: 'user123',
    action: 'login',
    timestamp: new Date().toISOString()
});

// Error handling example
try {
    throw new Error('Something went wrong');
} catch (error) {
    logger.error({
        msg: 'Error occurred',
        error: error.message,
        stack: error.stack
    });
}
```

### Step 3. Containerize the application

Create a Dockerfile for your application:
```Dockerfile
# Dockerfile
FROM node:16-alpine

# Create app directory
WORKDIR /usr/src/app

# Install dependencies
COPY package*.json ./
RUN npm install

# Copy application code
COPY tracing.js ./
COPY app.js ./

# Set environment variables (can be overridden in Kubernetes deployment)
ENV OTEL_EXPORTER_OTLP_LOGS_PROTOCOL=grpc
ENV OTEL_EXPORTER_OTLP_LOGS_ENDPOINT=http://otel-collector:4317
ENV OTEL_RESOURCE_ATTRIBUTES=service.name=pino-service,service.version=1.0.0,deployment.environment=production

# Command to run the application
CMD ["node", "app.js"]
```

### Step 2. Run with Docker

```bash
docker build -t my-pino-app .
docker run -d my-pino-app
```

### Step 3. Deploy to Kubernetes (Optional)

* Create a deployment and add the environment variables using either ConfigMap or Secrets based on your security requirements
```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pino-logger
  labels:
    app: pino-logger
spec:
  replicas: 1
  selector:
    matchLabels:
      app: pino-logger
  template:
    metadata:
      labels:
        app: pino-logger
    spec:
      containers:
      - name: pino-logger
        image: your-registry/your-image:latest
        imagePullPolicy: Always
        env:
        - name: OTEL_EXPORTER_OTLP_LOGS_PROTOCOL
          value: 'grpc'
        - name: OTEL_EXPORTER_OTLP_LOGS_ENDPOINT
          value: 'http://<service-name>.<namespace>.svc.cluster.local:4317'
        - name: OTEL_RESOURCE_ATTRIBUTES
          value: 'service.name=pino-logger,service.version=1.0.0'

```
