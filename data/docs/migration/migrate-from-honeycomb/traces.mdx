## Data Model Comparison

### Honeycomb: [Event-Centric Model](https://docs.honeycomb.io/investigate/analyze/explore-traces/)
- **Structure**: Traces as collections of events with arbitrary key-value pairs
- **Schema**: Flexible - any attribute can be added to any event
- **Focus**: Optimized for high-cardinality analysis
- **Relationships**: Parent-child via `trace_id` and `parent_id`

### SigNoz: [Span-Centric Model](https://signoz.io/blog/distributed-tracing-span/)
- **Structure**: Spans are fundamental building blocks of distributed tracing following OpenTelemetry specification
- **Schema**: Standardized OpenTelemetry semantic conventions
- **Focus**: Hierarchical span relationships with [span context](https://signoz.io/blog/opentelemetry-context-propagation/)
- **Relationships**: SpanContext carries key identifying details (trace ID, span ID, trace flags) across process boundaries

## Trace Structure Examples

### Honeycomb Event Format
```json
{
  "trace.trace_id": "abc123",
  "trace.span_id": "def456", 
  "trace.parent_id": "ghi789",
  "service.name": "api-service",
  "http.method": "GET",
  "http.url": "/api/users",
  "duration_ms": 45,
  "http.status_code": 200
}
```

### SigNoz Span Format
```json
{
  "traceId": "abc123",
  "spanId": "def456",
  "parentSpanId": "ghi789",
  "operationName": "HTTP GET /api/users",
  "startTime": "2024-01-01T12:00:00Z",
  "duration": 45000000,
  "tags": {
    "http.method": "GET",
    "http.url": "/api/users",
    "http.status_code": 200
  },
  "process": {
    "serviceName": "api-service"
  }
}
```

- **[Bubble Charts](https://docs.honeycomb.io/investigate/analyze/detect-anomalies/)**: Anomaly detection with outlier highlighting
- **[Heatmaps](https://docs.honeycomb.io/investigate/analyze/visualize-events/)**: Latency distribution patterns
- **[BubbleUp](https://www.honeycomb.io/bubbleup)**: Automatic anomaly detection

- **[Flame Graph](https://signoz.io/blog/flamegraphs/)**: Hierarchical representation showing nested span relationships
- **[Service Map](https://signoz.io/docs/userguide/service-map/)**: Visual representation of service interactions and calls
- **[Trace Timeline](https://signoz.io/docs/userguide/span-details/)**: Gantt chart visualization of trace execution

## Migration Strategies

### 1. Attribute Mapping

| Honeycomb Field | SigNoz Field | Notes |
|:---------------|:-------------|:------|
| `trace.trace_id` | `traceId` | Direct mapping |
| `trace.span_id` | `spanId` | Direct mapping |
| `trace.parent_id` | `parentSpanId` | Direct mapping |
| `service.name` | `process.serviceName` | Moved to process level |
| `duration_ms` | `duration` | Convert to nanoseconds |

### 2. Query Pattern Translation

#### Error Rate Analysis
```yaml
# Honeycomb
COUNT() WHERE http.status_code >= 500 / COUNT() * 100

# SigNoz
Filter: http.status_code >= 500
Metric: Error rate percentage
```

#### Latency Percentiles
```yaml
# Honeycomb
P95(duration_ms) WHERE service.name = 'api-service'

# SigNoz
Service: api-service
Aggregation: P95 latency
```

### 3. Instrumentation Updates

#### Python Example
```python
# Honeycomb
import beeline
beeline.add_field("user.id", "user123")
beeline.add_field("business.tier", "premium")

# SigNoz (OpenTelemetry)
from opentelemetry import trace
span = trace.get_current_span()
span.set_attribute("user.id", "user123")
span.set_attribute("business.tier", "premium")
```

```sql

COUNT() WHERE service.name = 'api' 
AND http.status_code >= 400 
GROUP BY user.tier, http.endpoint
```

```yaml

  - http.status_code >= 400
Group by: user.tier, http.route
```

## Migration Validation Steps

### Data Validation
- **Trace Completeness**: Verify all spans are captured
- **Timing Accuracy**: Compare duration and sequence in waterfall charts
- **Service Dependencies**: Validate service interactions in service map
- **Error Traces**: Ensure error spans are properly identified

### Query Validation
- **Latency Percentiles**: Compare P95/P99 values
- **Error Rates**: Validate error percentage calculations
- **Throughput**: Verify request count accuracy
- **Service Performance**: Check service-level metrics

## Common Migration Challenges

### 1. High-Cardinality Analysis
**Challenge**: Honeycomb's strength in arbitrary attribute combinations  
**Solution**: Use SigNoz's trace filtering and attribute-based grouping

### 2. BubbleUp Replacement
**Challenge**: No automatic anomaly detection  
**Solution**: Create custom dashboards with outlier detection using percentile comparisons

### 3. Flexible Schema
**Challenge**: Honeycomb's arbitrary field addition  
**Solution**: Use OpenTelemetry's structured attributes with semantic conventions

### 4. Team Workflow Changes
**Challenge**: Different debugging approaches  
**Solution**: Train team on flame graphs for identifying performance bottlenecks
